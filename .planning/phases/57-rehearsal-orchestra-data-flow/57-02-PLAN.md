<objective>
Add cascade deactivation from orchestra to its rehearsals, ensuring no orphan references when an orchestra is deleted/deactivated.

Purpose: Currently, removeOrchestra sets `isActive: false` on the orchestra but does NOT touch its rehearsals. This leaves rehearsals discoverable and linked to a deactivated orchestra — a data integrity bug. The fix wraps the entire removeOrchestra operation in a transaction that also deactivates all linked rehearsals.

Output: orchestra.service.js with transactional cascade deactivation in removeOrchestra.
</objective>

   Note: rehearsals currently use hard-delete pattern (no isActive field). However, for cascade deactivation, we should SOFT-DELETE rehearsals (set isActive: false) to preserve historical data.    Actually, looking at the rehearsal service again, rehearsals are HARD deleted (no isActive field, no soft-delete pattern). To keep consistency with the existing rehearsal deletion pattern AND to satisfy RORCH-02 ("removes or archives all associated rehearsals"), we should HARD DELETE the rehearsals and clear the orchestra's rehearsalIds array. This is consistent with how removeRehearsal works (findOneAndDelete).

   Revised approach:
   ```js
   // Delete all rehearsals for this orchestra
   const rehearsalCollection = await getCollection('rehearsal')
   const rehearsals = await rehearsalCollection.find(
     { groupId: orchestraId, tenantId },
     { session }
   ).toArray()
   const rehearsalIds = rehearsals.map(r => r._id.toString())

   await rehearsalCollection.deleteMany(
     { groupId: orchestraId, tenantId },
     { session }
   )

   // Delete associated attendance records for those rehearsals
   if (rehearsalIds.length > 0) {
     const activityCollection = await getCollection('activity_attendance')
     await activityCollection.deleteMany(
       { sessionId: { $in: rehearsalIds }, activityType: 'תזמורת', tenantId },
       { session }
     )
   }
   ```
