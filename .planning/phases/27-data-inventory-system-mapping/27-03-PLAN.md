Create `ARCHITECTURE-DIAGRAM.md` as a formal compliance document. Structure:

1. **Document header:** Title ("System Architecture and Security Classification"), version 1.0, date, classification "INTERNAL -- COMPLIANCE DOCUMENT".

2. **Overview:** Paragraph describing the Tenuto.io platform as a multi-tenant SaaS application for Israeli music conservatory management. Summarize the technology stack: Node.js + Express REST API, React 18 SPA frontend, MongoDB Atlas database, AWS S3 file storage, hosted on Render.

3. **Component diagram (Mermaid):** Create a Mermaid `graph TD` (or `flowchart TD`) diagram showing all platform components with classification labels on each node. Components to include:

   - **Browser** [PUBLIC] -- User's web browser
   - **React Frontend SPA** [PUBLIC] -- Served from Render as static files. Contains no sensitive data itself (data fetched via API).
   - **Express API Server** [SENSITIVE] -- Hosted on Render. Processes all data categories. Contains middleware chain for auth, tenant isolation, and RBAC.
   - **MongoDB Atlas** [RESTRICTED] -- All collections including minors' data and credentials. Hosted on MongoDB Atlas cloud.
   - **AWS S3** [SENSITIVE] -- eu-central-1 (Frankfurt). Stores uploaded documents (bagrut files: PDF, DOC, JPG, PNG).
   - **SendGrid** [SENSITIVE] -- US-based. Transactional email (invitations, password resets). Receives teacher names and email addresses.
   - **Gmail Fallback** [SENSITIVE] -- Google infrastructure. Fallback email delivery. Same data as SendGrid.
   - **Pino Logger** [INTERNAL] -- Server-local stdout captured by Render. Structured JSON logs with sensitive field redaction.

   Per user decision: Data classification labels appear directly on diagram nodes (e.g., the node text includes `[RESTRICTED]`).

   Show connections between components:
   - Browser <--> React Frontend (HTTPS)
   - React Frontend <--> Express API (HTTPS, JWT auth)
   - Express API <--> MongoDB Atlas (TLS, MongoDB wire protocol)
   - Express API <--> AWS S3 (HTTPS, AWS SDK)
   - Express API --> SendGrid (HTTPS API)
   - Express API --> Gmail (SMTP TLS)
   - Express API --> Pino Logger (stdout)

4. **Component details table:** Below the diagram, a table with columns:
   Component | Technology | Hosting Provider | Data Classification | Data Types Processed | Security Controls

   Fill in for each component from research section 2.1 and 2.3. Security controls include: helmet, express-mongo-sanitize, rate-limit, CORS allowlist, JWT auth, tenant isolation, bcrypt password hashing, cookie-based refresh tokens, etc.

5. **Security boundary annotations:** A text section describing the security boundaries:
   - External boundary: Browser to Render (HTTPS/TLS)
   - Internal boundary: Render to MongoDB Atlas (TLS)
   - Internal boundary: Render to AWS S3 (HTTPS with access keys)
   - Cross-border boundary: Render to SendGrid/Gmail (data exits to US/Google)
   - No VPN or private networking between components (all communicate over public internet with TLS)

6. **Middleware security chain diagram:** A second small Mermaid diagram showing the Express middleware pipeline:

  <name>Task 2: Create DATA-FLOW-MAP.md with Mermaid data flow diagram</name>
  <files>.planning/compliance/DATA-FLOW-MAP.md</files>
  <action>
Create `DATA-FLOW-MAP.md` as a formal compliance document. Structure:

1. **Document header:** Title ("Personal Data Flow Map"), version 1.0, date, classification "INTERNAL -- COMPLIANCE DOCUMENT".

2. **Overview:** Paragraph explaining this document traces the movement of personal data through the platform, classified by sensitivity level. Purpose: identify every path personal data takes from collection to storage to potential deletion.

3. **Data flow diagram (Mermaid):** Create a Mermaid `flowchart LR` (left-to-right) diagram showing personal data flows. Per user decision: arrows labeled with classification CATEGORIES (e.g., "SENSITIVE data", "RESTRICTED data"), NOT specific field names.

   Key flows to diagram:

   **Authentication flow:**
   - Browser -->|"RESTRICTED data (credentials)"| Express API -->|"RESTRICTED data"| MongoDB Atlas (teacher collection)
   - Express API -->|"RESTRICTED (JWT)"| Browser (stored in localStorage)

   **Student data flow:**
   - Browser -->|"RESTRICTED data (student PII)"| Express API -->|"RESTRICTED data"| MongoDB Atlas (student collection)
   - Express API -->|"RESTRICTED data (grades)"| MongoDB Atlas (bagrut collection)

   **Teacher data flow:**
   - Browser -->|"SENSITIVE data (teacher PII)"| Express API -->|"SENSITIVE data"| MongoDB Atlas (teacher collection)

   **Import flow:**
   - Browser -->|"RESTRICTED + SENSITIVE data (Excel file)"| Express API -->|"Parsed data"| MongoDB Atlas (import_log) -->|"Executed records"| MongoDB Atlas (target collection)

   **Export flow:**
   - MongoDB Atlas -->|"RESTRICTED + SENSITIVE data"| Express API -->|"XLSX file"| Browser
   - Express API -->|"Snapshot data"| MongoDB Atlas (ministry_report_snapshots)

   **Email flow:**
   - Express API -->|"SENSITIVE data (name, email, token URL)"| SendGrid/Gmail -->|"Email"| Teacher inbox

   **File upload flow:**
   - Browser -->|"SENSITIVE data (documents)"| Express API -->|"SENSITIVE data"| AWS S3

   **Deletion flow:**
   - Express API -->|"RESTRICTED + SENSITIVE data (full document snapshot)"| MongoDB Atlas (deletion_snapshots)
   - Express API -->|"ALL data categories"| MongoDB Atlas (tenant_deletion_snapshots) [tenant purge]

   **Logging flow:**
   - Express API -->|"INTERNAL data (redacted logs)"| Pino/stdout (Render captures)
   - Express API -->|"INTERNAL data (audit events)"| MongoDB Atlas (platform_audit_log)

4. **Flow details table:** For each flow, a table with columns:
   Flow Name | Source | Destination | Data Classification | Transport Security | Authentication | Notes

5. **Data at rest summary:** Table showing where personal data ultimately resides:
   Storage Location | Data Classifications Present | Encryption at Rest | Access Control

   - MongoDB Atlas: RESTRICTED + SENSITIVE + INTERNAL. Encryption: MongoDB Atlas default encryption at rest. Access: Connection string auth + tenant isolation.
   - AWS S3: SENSITIVE. Encryption: S3 default encryption. Access: AWS access key auth.
   - Render (logs): INTERNAL (sensitive fields redacted by Pino). Encryption: Render platform default. Access: Render dashboard auth.
   - Browser (localStorage): RESTRICTED (JWT token). Encryption: None (browser storage). Access: Same-origin policy.

6. **Data in transit summary:** All external connections use TLS/HTTPS. Note that JWT tokens travel in Authorization headers over HTTPS. Note cross-border transfer to SendGrid (US).

7. **Key risks identified from flow analysis:** Brief list:
   - JWT stored in localStorage (vulnerable to XSS)
   - Full document snapshots in deletion_snapshots have no separate encryption
   - Import preview data persisted with no cleanup
   - Cross-border email data transfer to US-based SendGrid
