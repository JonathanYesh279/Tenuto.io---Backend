  <action>
Create the `.planning/compliance/` directory.

Create `DATA-INVENTORY.md` as a formal Database Definition Document. Structure:

1. **Document header:** Title, version, date, classification ("INTERNAL -- COMPLIANCE DOCUMENT"), document owner (placeholder for security officer to be appointed in Phase 28).

2. **Document purpose:** One paragraph explaining this document catalogs all personal data holdings per Israeli Privacy Protection Regulations (Information Security), 2017. Security level: MEDIUM.

3. **Classification scheme:** Define the four sensitivity tiers with descriptions:
   - PUBLIC: No privacy implications (instrument lists, enum constants, health check data)
   - INTERNAL: Operational data not publicly visible (tenant config, school year settings, schedule data)
   - SENSITIVE: Adult personal data (teacher PII, contact info, ID numbers, organizational contact details)
   - RESTRICTED: Highest classification -- minors' data (student PII for under-18s) AND credentials (passwords, tokens, API keys). Per user decision: minors' data gets the highest classification level alongside credentials.

4. **Collection summary table:** A single overview table listing all 21 collections with columns: Collection Name | Tenant-Scoped | Contains Personal Data | Overall Classification | Record Count Estimate.

   Tenant-scoped collections (15): teacher, student, orchestra, rehearsal, theory_lesson, bagrut, school_year, activity_attendance, hours_summary, import_log, ministry_report_snapshots, deletion_audit, deletion_snapshots, security_log.

   Platform-level collections (7): tenant, super_admin, platform_audit_log, tenant_deletion_snapshots, migration_backups, integrityAuditLog, integrityStatus, healthcheck.

   Note: Record count is "N/A -- runtime data" for all (we cannot query production).

5. **Detailed field-level inventory:** For EACH of the 21 collections, create a section with:
   - Section heading: `### {collection_name} Collection` with overall classification badge
   - Source reference (which code file defines the schema)
   - A markdown table with columns: Field Path | Type | Sensitivity | Purpose
   - Every field listed individually (per user decision -- no grouping of related fields)
   - Use the exact field data from 27-RESEARCH.md sections 1.2 through 1.4

   For the `teacher` collection: include ALL fields from research including the 11 managementInfo.*Hours fields (list each: coordinationHours, guidanceHours, managementHours, departmentManagementHours, professionalDevelopmentHours, preparationHours, welfareHours, otherHours, totalManagementHours, schoolManagementHours, classManagementHours). Include ALL credentials.* fields individually.

   For the `student` collection: include ALL personalInfo, academicInfo, enrollments, and teacherAssignments sub-fields individually.

   For collections where research shows "Various | Various | internal" (integrityAuditLog, integrityStatus), document what is known and note "schema details inferred from collection purpose."

   IMPORTANT: Use the exact sensitivity classifications from the research. Do NOT downgrade restricted fields to sensitive or vice versa. Minors' data fields = RESTRICTED. Adult PII = SENSITIVE. Credentials = RESTRICTED.

6. **Notes section:** Add a "Data Classification Notes" section at the bottom documenting:
   - Why minors' data is classified as RESTRICTED (same level as credentials) -- per Israeli privacy regulations and user decision
   - The denormalization concern: student names appear in teacher.teaching.timeBlocks[].assignedLessons[].studentName
   - Collections that contain embedded PII in blob fields: import_log.previewData, ministry_report_snapshots.snapshotData, deletion_snapshots.snapshotData, tenant_deletion_snapshots.collectionSnapshots

Document must be written in English. No Hebrew glossary in this document (that is a separate deliverable).
