Create `DATA-PURPOSES.md` as a formal compliance document. Structure:

1. **Document header:** Title ("Data Processing Purposes and Retention Policy"), version 1.0, date, classification "INTERNAL -- COMPLIANCE DOCUMENT".

2. **Regulatory context:** One paragraph referencing Israeli Privacy Protection Regulations 2017, explaining that each data processing activity must have a lawful basis and defined retention period.

3. **Lawful basis types:** Define the four basis types used:
   - Consent: Data subject has given explicit consent
   - Contractual necessity: Required to deliver music education services per enrollment agreement
   - Legal obligation: Required by Israeli law (e.g., Ministry of Education reporting, tax/employment records)
   - Legitimate interest: Necessary for platform operation where interest does not override data subject rights

4. **Collection-by-collection table:** Create a detailed table for ALL 21 collections with columns:
   Collection | Data Purpose | Lawful Basis | Rationale | Contains PII | Retention: Current State | Retention: Recommended | Action Needed

   Use lawful basis from research section 1.4. For each collection:
   - Data Purpose: One sentence describing what the collection is for
   - Lawful Basis: One of the four types
   - Rationale: One-line explanation (per user decision format)
   - Contains PII: Yes/No
   - Retention Current State: "No retention policy (indefinite)" for all collections (research confirms no TTL indexes exist)
   - Retention Recommended: Suggest a recommendation (e.g., "7 years for legal compliance", "90 days after import execution" for import_log, "Archive after school year close" for operational data) -- per user decision, these are RECOMMENDATIONS not binding periods
   - Action Needed: Flag as "NEEDS RETENTION POLICY" for collections with PII, "None" for public/operational

5. **Retention gap analysis:** A dedicated section listing all collections where no retention enforcement exists (all of them currently), prioritized by risk:
   - CRITICAL: Collections with RESTRICTED data and no retention (deletion_snapshots, tenant_deletion_snapshots, import_log with previewData)
   - HIGH: Collections with SENSITIVE data and no retention (teacher, tenant, migration_backups)
   - MEDIUM: Operational collections that accumulate indefinitely (ministry_report_snapshots, platform_audit_log)

6. **Cross-border data transfer note:** Flag that SendGrid transfers teacher email data to the US. Document that this requires either Standard Contractual Clauses or equivalent cross-border transfer basis under Israeli regulations.

Per user decision: Do NOT define binding retention periods -- only recommend for later decision.

**MINORS-DATA.md** -- Formal document identifying all minors' data in the platform. Structure:

1. **Document header:** Title ("Minors' Data Protection Assessment"), version 1.0, date, classification "INTERNAL -- COMPLIANCE DOCUMENT".

2. **Regulatory context:** Explain that students in Israeli music conservatories are primarily school-age children (grades alef through yud-bet, approximately ages 6-18). Under Israeli Privacy Protection Regulations, minors' personal data requires heightened protection. Note that parental/guardian consent is required for processing minors' data.

3. **Minors' data inventory:** List EVERY location where minors' data appears, organized by collection:

   **Primary collections (direct minors' data):**
   - `student` collection: All personalInfo fields (firstName, lastName, phone, age, address, parentName, parentPhone, parentEmail, studentEmail), academic records, enrollment data. Classification: RESTRICTED.
   - `bagrut` collection: Examination grades, detailed grading breakdowns, director evaluations, final grades. References student by studentId. Classification: RESTRICTED.

   **Secondary collections (indirect minors' data via references):**
   - `teacher` collection: `teaching.timeBlocks[].assignedLessons[].studentName` denormalizes minor's name. Classification of this field: RESTRICTED.
   - `orchestra` collection: `memberIds[]` references student documents (IDs only, no PII directly).
   - `rehearsal` collection: `attendance.present[]` and `attendance.absent[]` contain student IDs.
   - `theory_lesson` collection: `studentIds[]`, `attendance.present[]`, `attendance.absent[]` contain student IDs.
   - `activity_attendance` collection: `studentId` references a student.

   **Snapshot collections (complete minors' data copies):**
   - `deletion_snapshots`: May contain full student document snapshots including all RESTRICTED fields.
   - `tenant_deletion_snapshots`: Contains complete tenant data dump including all student records.
   - `import_log`: `previewData` may contain parsed student data from Excel imports.
   - `ministry_report_snapshots`: `snapshotData` contains aggregated student data.

4. **Special handling requirements:** For each category above, document:
   - Access controls currently in place (tenant isolation, role-based access, buildScopedFilter for teachers)
   - Gap: No specific minors' data access logging exists
   - Gap: No parental consent mechanism is implemented
   - Gap: No age verification at data entry
   - Gap: Snapshot collections retain minors' data indefinitely without separate controls
   - Recommendation: v1.6 should implement minors' data access audit trail

5. **Denormalization risk:** Dedicated subsection on the `studentName` field in teacher documents -- this means deleting a student record does not remove their name from teacher schedule blocks. Flag for v1.6 remediation.

---

**DATA-MINIMIZATION.md** -- Annual review process definition. Structure:

1. **Document header:** Title ("Data Minimization Review Process"), version 1.0, date.

2. **Purpose:** Define the annual process for reviewing data holdings to ensure only necessary data is collected and retained, per data minimization principle.

3. **Review scope:** List what the annual review covers:
   - All MongoDB collections and their fields
   - Retention compliance (are recommended retention periods being enforced?)
   - Data that is no longer needed (inactive students, closed school years, expired tokens)
   - Snapshot collections (deletion_snapshots, tenant_deletion_snapshots, migration_backups) -- are they being purged?
   - Import log cleanup (is previewData being removed after execution?)
   - Denormalized data freshness (is studentName in teacher docs synchronized?)

4. **Review process:** Step-by-step procedure:
   - Step 1: Security Officer (to be appointed in Phase 28) initiates annual review
   - Step 2: Run data inventory validation against current codebase (check for new collections/fields)
   - Step 3: Review each collection against its documented purpose -- is any data collected that is not needed?
   - Step 4: Check retention compliance -- are TTL indexes and cleanup jobs running?
   - Step 5: Review access logs for anomalous patterns (bulk exports, unusual access to minors' data)
   - Step 6: Document findings and remediation actions
   - Step 7: Update DATA-INVENTORY.md and DATA-PURPOSES.md if data holdings have changed

5. **Review schedule:** Annual, triggered on the anniversary of the initial review. Interim reviews triggered by: new collection creation, new third-party vendor integration, data breach incident, regulatory change.

6. **Review output template:** A simple checklist template the reviewer fills out (markdown checklist format).
