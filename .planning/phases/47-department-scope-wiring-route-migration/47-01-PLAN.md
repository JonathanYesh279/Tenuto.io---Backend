  <name>Task 1: Wire req.permissionScope through all controllers and services to buildScopedFilter</name>
  <files>
    api/student/student.controller.js
    api/student/student.service.js
    api/teacher/teacher.controller.js

    api/bagrut/bagrut.service.js
  </files>
  <action>
    For each of the 6 controller/service pairs (student, teacher, orchestra, rehearsal, theory, bagrut), apply the same mechanical pattern:

    **Controllers** -- pass `req.permissionScope` to service via options:
    - In every controller function that calls a service list method (getStudents, getTeachers, getOrchestras, etc.), add `scope: req.permissionScope` to the options object.
    - Example: `{ context: req.context }` becomes `{ context: req.context, scope: req.permissionScope }`
    - For getStudentById in student.controller.js, also pass scope to `canAccessStudent(id, req.context, req.permissionScope)` (it already accepts a 3rd arg per Phase 40).
    - Do NOT change write operations (add/update/delete) -- scope only affects reads.

    **Services** -- forward scope to buildScopedFilter as 4th argument:
    - In each service's list function (getStudents, getTeachers, etc.), destructure `scope` from options alongside `context`.
    - Pass scope as the 4th argument to buildScopedFilter: `buildScopedFilter('student', {}, context, scope)`
    - Current pattern: `buildScopedFilter('student', {}, context)` -- add `, scope` after context.

    Specific files and their buildScopedFilter call sites:

    1. **student.service.js** line ~35: `buildScopedFilter('student', {}, context)` -> `buildScopedFilter('student', {}, context, scope)`
       - Destructure: `const { context, scope } = options;`
    2. **teacher.service.js** line ~58: `buildScopedFilter('teacher', _buildCriteria(filterBy), context)` -> add `, scope`
       - Destructure: `const { context, scope } = options;`
    3. **orchestra.service.js** line ~28: `buildScopedFilter('orchestra', _buildCriteria(filterBy), context)` -> add `, scope`
       - Destructure: `const { context, scope } = options;`
    4. **rehearsal.service.js** line ~43: `buildScopedFilter('rehearsal', _buildCriteria(filterBy), context)` -> add `, scope`
       - Destructure: `const { context, scope } = options;`
    5. **theory.service.js** line ~67: `buildScopedFilter('theory_lesson', createLessonFilterQuery(filterBy), context)` -> add `, scope`
       - Destructure: `const { context, scope } = options;`
    6. **bagrut.service.js** line ~56: `buildScopedFilter('bagrut', _buildCriteria(filterBy), context)` -> add `, scope`
       - Destructure: `const { context, scope } = options;`

    **Backward compatibility note:** When scope is undefined (callers that don't pass it), buildScopedFilter defaults to null which triggers legacy isAdmin-based behavior. No breakage.
  </action>
  <verify>

    Run: `grep -rn "req.permissionScope" api/ --include="*.controller.js"` -- should return results in all 6 controller files.

<task type="auto">
  <name>Task 2: Add requirePermission to analytics and schedule attendance routes</name>
  <files>
    api/analytics/attendance.routes.js

  <action>
    Both attendance route files currently have NO per-route permission middleware. They rely only on the server.js-level authenticateToken + buildContext chain. Add requirePermission to each route.

    **api/analytics/attendance.routes.js** (7 routes):
    - Import: `import { requirePermission } from '../../middleware/auth.middleware.js';`
    - Student attendance stats (GET /students/:studentId/attendance): `requirePermission('students', 'view')` -- viewing student data
    - Teacher attendance analytics (GET /teachers/:teacherId/attendance): `requirePermission('reports', 'view')` -- analytics is reporting
    - Overall attendance report (GET /attendance/overall): `requirePermission('reports', 'view')` -- aggregate reporting
    - Attendance trends (GET /attendance/trends): `requirePermission('reports', 'view')` -- analytics
    - Attendance comparison (POST /attendance/compare): `requirePermission('reports', 'view')` -- read-only comparison
    - Generate insights (GET /:entityType/:entityId/insights): `requirePermission('reports', 'view')` -- analytics
    - Export attendance report (POST /attendance/export): `requirePermission('reports', 'export')` -- exporting

    **api/schedule/attendance.routes.js** (3 routes):
    - Import: `import { requirePermission } from '../../middleware/auth.middleware.js';`
    - Student private lesson stats (GET /students/:studentId/private-lesson-attendance): `requirePermission('students', 'view')` -- student data
    - Student attendance history (GET /students/:studentId/attendance-history): `requirePermission('students', 'view')` -- student data
    - Teacher attendance overview (GET /teachers/:teacherId/lesson-attendance-summary): `requirePermission('schedules', 'view')` -- schedule data for teacher self-view

    Domain rationale: Student-specific attendance routes use 'students' domain (consistent with student.route.js). Analytics/aggregate routes use 'reports' domain (consistent with Phase 41's past-activities pattern). Teacher schedule attendance uses 'schedules' domain (consistent with Phase 41's hours-summary self-view pattern).
