Create a user notification policy document following the established Phase 27-29 format (document ID, version, classification header, cross-references, review schedule).

**Document structure:**

1. **Header block:** Document ID: LOG-01, Version 1.0, Date 2026-03-02, Classification INTERNAL -- COMPLIANCE DOCUMENT, Owner: Security Officer (SECOFF-01/02), Maintained By: Development Team, Review Cycle: Annual or upon changes to monitoring capabilities, Related Documents: ACPOL-03 (ACCESS-LOGGING-POLICY.md), SECOFF-01/02 (SECURITY-OFFICER.md), PERS-02 (PERSONNEL-SECURITY.md training), GLOSS-01 (GLOSSARY.md).

2. **Purpose (Section 1):** Reference Israeli Privacy Protection Regulations Regulation 10(e) -- "A database controller will inform the database authorized users of the monitoring mechanism to the database systems." This document satisfies the transparency requirement by informing all authorized users that their access and activity are logged and monitored. Note that ACCESS-LOGGING-POLICY.md (ACPOL-03) Section 8 explicitly identified this gap and deferred the full notification framework to Phase 30.

3. **Notification Scope (Section 2):** Define who receives the notification:
   - Platform personnel (super admins, developers, Security Officer) -- all infrastructure and application access
   - Tenant personnel (conservatory admins, teachers) -- all application access within their tenant scope
   - The notification covers ALL authorized user categories defined in ACCESS-CONTROL-POLICY.md (ACPOL-01)

4. **Current Monitoring Capabilities (Section 3):** This is the CRITICAL section -- must be honest.
   - **What IS currently logged** (reference ACPOL-03 Section 2.1):
     - Super admin audit log: login, impersonation, tenant operations (in admin_audit_log collection)
     - Deletion audit: all cascade deletion operations with snapshots (in deletion_snapshots collection)
     - Security events: login attempts, token operations, critical auth events (in security_log collection -- limited scope)
     - Application errors: server-side errors (Render application logs, 30-day retention)
   - **What is NOT currently logged** (reference ACPOL-03 Section 3.1):
     - Individual record access (student/teacher data reads)
     - Data export operations
     - Report generation
     - Search queries
     - Session duration and activity patterns
   - **Planned enhancements (v1.6):** Comprehensive access logging covering all data operations, anomaly detection, real-time alerting

5. **Notification Text (Section 4):** Provide a draft notification text in English. The notification must:
   - Inform users their access and activity MAY be logged and monitored
   - Use "may be" language (not "is actively monitored") for capabilities not yet implemented
   - State the purpose: security and regulatory compliance, not performance evaluation
   - State that monitoring data is retained per the retention schedule in ACPOL-03 Section 5
   - Provide Security Officer contact for questions
   - Include a note that Hebrew translation is required before production deployment
   - Keep language non-technical and accessible to conservatory teachers

   Example structure:
   > "Your access to and activity within Tenuto.io may be logged and monitored for security and regulatory compliance purposes. This includes login events, administrative actions, and data operations. Monitoring data is retained in accordance with our data retention policy. For questions about monitoring practices, contact the Security Officer at [CONTACT]. By continuing to use the platform, you acknowledge this monitoring."

6. **Notification Delivery Mechanism (Section 5):** Define the policy-level approach (not the technical implementation):
   - Recommended primary method: Login banner displayed upon first login after policy deployment, and periodically (e.g., annually at school year start)
   - Alternative method: Terms of Service / Acceptable Use Policy clause
   - Supplementary: Reference in security awareness training (PERS-02 Topic 1)
   - Implementation deferred to v1.6; for pre-launch, include in written onboarding materials (PERS-01)

7. **User Rights Regarding Monitored Data (Section 6):**
   - Users may request information about what data is logged about their activity
   - Users may request correction of inaccurate monitoring data
   - Requests directed to Security Officer per SECOFF-01/02
   - Response timeline: within 30 days per Israeli Privacy Protection Law

8. **Monitoring Data Retention (Section 7):** Reference ACPOL-03 Section 5 retention schedule:
   - Operational logs: 30 days
   - Security logs: 2 years
   - Legal/compliance logs: 7 years
   - Minors' data access logs: 7 years (when implemented)

9. **Review Schedule (Section 8):** Annual review or upon significant changes to monitoring capabilities (e.g., after v1.6 technical hardening). The notification text must be updated whenever new monitoring categories are activated.

10. **Document Cross-References (Section 9):** Table of related documents.

**Anti-patterns to avoid:**
- Do NOT claim comprehensive monitoring exists. The notification must match the CURRENT state documented in ACPOL-03 Section 2.1.
- Do NOT reference monitoring capabilities documented as "NOT LOGGED" or "PLANNED for v1.6" in ACPOL-03 Section 3.1 as if they exist today.
- Do NOT use legalistic language that conservatory teachers would not understand.

Create a mobile device policy document following the established Phase 27-29 format.

**Document structure:**

1. **Header block:** Document ID: MOB-01, Version 1.0, Date 2026-03-02, Classification INTERNAL -- COMPLIANCE DOCUMENT, Owner: Security Officer (SECOFF-01/02), Maintained By: Development Team, Review Cycle: Annual or upon platform architecture changes, Related Documents: ACPOL-01 (ACCESS-CONTROL-POLICY.md), ACPOL-02 (AUTH-POLICY.md), PERS-01/02/03 (PERSONNEL-SECURITY.md), RISK-01 (RISK-ASSESSMENT.md), SMAP-01 (ARCHITECTURE-DIAGRAM.md).

2. **Purpose (Section 1):** Reference Israeli Privacy Protection Regulations Regulation 12 -- restrict or deny portable device connections "in a manner compatible with the information security level applicable to the database, and employing commonly used encryption methods." This document defines usage restrictions and required safeguards for accessing Tenuto.io from any device.

3. **Scope (Section 2):**
   - Applies to ALL devices used to access Tenuto.io: smartphones, tablets, laptops, desktop computers
   - Applies to ALL user types: platform personnel (super admins, developers) and tenant personnel (admins, teachers)
   - Tenuto.io is a browser-based SaaS platform. There is NO native mobile app, NO MDM enrollment, NO device-level management. All access is through standard web browsers.
   - This policy focuses on user behavior and browser hygiene, not device-level technical controls.

4. **Permitted Access Methods (Section 3):**
   - Access is permitted ONLY through web browsers (Chrome, Firefox, Safari, Edge -- current versions)
   - No native application exists -- do not install any third-party app claiming to provide Tenuto.io access
   - No API access from personal devices (API access is server-to-server only)

5. **Network Security Requirements (Section 4):**
   - REQUIRED: Use trusted networks (home, school, cellular data)
   - AVOID: Public Wi-Fi (cafes, airports, hotels) -- if unavoidable, use cellular data instead
   - All connections use TLS 1.2+ encryption (enforced by platform, reference SMAP-01 Section 4)
   - Risk note: Public networks increase exposure to man-in-the-middle attacks despite TLS (reference R-04 JWT compromise scenario in RISK-01)

6. **Device Security Requirements (Section 5):**
   - Screen lock REQUIRED (PIN, pattern, biometric, or password) -- auto-lock after 5 minutes of inactivity
   - Operating system updates MUST be applied promptly
   - Do NOT root/jailbreak devices used to access the platform
   - Do NOT disable device encryption (all modern iOS and Android devices encrypt by default)
   - For platform personnel: follow organization device security policy if stricter

7. **Browser Security Requirements (Section 6):**
   - Do NOT save Tenuto.io credentials in browser password managers on shared/public devices
   - Clear browser cache and cookies after using shared devices
   - Use private/incognito mode when accessing from shared devices
   - Keep browser updated to latest version
   - Log out explicitly when finished (do not rely on session timeout alone)
   - Note: JWT is stored in browser localStorage (reference ACPOL-02 and R-04 in RISK-01) -- clearing browser data removes the session

8. **Data Handling Restrictions (Section 7):**
   - Do NOT take screenshots of student personal data (grades, personal information, contact details)
   - Do NOT export or download student data to personal storage (Google Drive, Dropbox, USB, etc.)
   - Do NOT forward student data via personal messaging (WhatsApp, Telegram, personal email)
   - Do NOT copy-paste student data to external applications
   - Reference PERS-02 Training Topic 5 (minors' data special handling) and Topic 3 (data handling principles)
   - Exception: Authorized data exports through the platform's built-in export functionality for ministry reporting are permitted

9. **BYOD Policy (Section 8):**
   - Tenant personnel (teachers, admins): Personal devices ARE expected and permitted. No device enrollment or MDM required.
   - Platform personnel (developers, super admins): Follow organizational device policy if one exists. Personal devices permitted for application access only (not infrastructure access -- infrastructure accessed from secured development workstations only).
   - No device inventory or registration required at this time.

10. **Lost or Stolen Device Procedure (Section 9):**
    - Immediately change Tenuto.io password from another device
    - Report to tenant admin (for tenant personnel) or Security Officer (for platform personnel)
    - Admin/Security Officer assesses need for token revocation (increment tokenVersion)
    - If device had saved credentials: treat as potential credential compromise per INCD-01 severity classification
    - Remotely wipe browser data if device management allows (user responsibility -- platform cannot enforce)

11. **Current Technical Controls (Section 10):**
    Current state of mobile-specific security:
    - TLS 1.2+ on all connections (enforced)
    - JWT with 15-minute access token expiry (per ACPOL-02)
    - No mobile-specific restrictions exist (no responsive design restrictions, no device fingerprinting, no geofencing)
    - localStorage used for token storage on all devices including mobile browsers (known gap)

12. **Gap Analysis and v1.6 Planned Enhancements (Section 11):**
    Follow the three-part pattern: Current State -> Gap -> Planned Remediation.
    - Gap: No session timeout on inactivity (user must explicitly log out or wait for 15-min token expiry)
    - Gap: No device fingerprinting or concurrent session limits
    - Gap: No ability to remotely terminate specific sessions
    - Gap: No data-loss prevention for screenshots or copy-paste
    - v1.6 planned: Session inactivity timeout, concurrent session management, enhanced token revocation

13. **Review Schedule (Section 12):** Annual review or upon platform architecture changes (e.g., if native app is developed).

14. **Document Cross-References (Section 13):** Table of related documents.

**Anti-patterns to avoid:**
- Do NOT include MDM, device enrollment, app containerization, or enterprise mobile management requirements. This is a browser-based SaaS.
- Do NOT mention "installing the app" -- there is no app.
- Do NOT require device registration or inventory for tenant personnel.

  <name>Task 3: Create ENCRYPTION-POLICY.md (ENC-01)</name>
  <files>.planning/compliance/ENCRYPTION-POLICY.md</files>
  <action>
Create an encryption policy document following the established Phase 27-29 format. This document consolidates encryption information from ARCHITECTURE-DIAGRAM.md and DATA-FLOW-MAP.md into the single authoritative encryption policy.

**Document structure:**

1. **Header block:** Document ID: ENC-01, Version 1.0, Date 2026-03-02, Classification INTERNAL -- COMPLIANCE DOCUMENT, Owner: Security Officer (SECOFF-01/02), Maintained By: Development Team, Review Cycle: Annual or upon changes to encryption infrastructure, Related Documents: SMAP-01 (ARCHITECTURE-DIAGRAM.md), SMAP-02 (DATA-FLOW-MAP.md), SECPR-01/02/03 (SECURITY-PROCEDURES.md), RISK-01 (RISK-ASSESSMENT.md), ACPOL-02 (AUTH-POLICY.md).

2. **Purpose (Section 1):** Reference Israeli Privacy Protection Regulations Regulation 14 -- "The transfer of information from the database through a public network or the Internet will be conducted by commonly used encryption methods." This document defines the encryption standards for all data in transit and at rest on the Tenuto.io platform. It is the single authoritative source for encryption standards; ARCHITECTURE-DIAGRAM.md and DATA-FLOW-MAP.md provide the evidence that these standards are met.

3. **Encryption Standards for Data in Transit (Section 2):**
   Create a comprehensive table of ALL connection paths with their encryption status, sourced from ARCHITECTURE-DIAGRAM.md Section 4 and DATA-FLOW-MAP.md Section 5. Table columns: Connection Path, Protocol, Minimum Version, Certificate Management, Status.

   Connections to document:
   - Browser to API (Render): HTTPS / TLS 1.2+ / Render-managed certificate / ENFORCED
   - API to MongoDB Atlas: TLS 1.2+ / Atlas-managed certificate / ENFORCED
   - API to AWS S3: HTTPS / TLS 1.2+ / AWS-managed certificate / ENFORCED
   - API to SendGrid: HTTPS / TLS 1.2+ / SendGrid-managed certificate / ENFORCED
   - API to Gmail SMTP: TLS / Gmail-managed certificate / ENFORCED
   - Browser WebSocket (Socket.io): WSS (TLS) / Render-managed certificate / ENFORCED
   - MongoDB Atlas internal replication: TLS / Atlas-managed / ENFORCED (managed by provider)

   Policy statement: ALL data transmission over public networks MUST use TLS 1.2 or higher. No unencrypted connections are permitted.

4. **Encryption Standards for Data at Rest (Section 3):**
   Table of ALL storage locations with encryption status, sourced from DATA-FLOW-MAP.md Section 4. Table columns: Storage Location, Encryption Method, Key Management, Status.

   Storage to document:
   - MongoDB Atlas (22 collections): AES-256 / Atlas-managed encryption at rest / ENFORCED (provider default)
   - AWS S3 (file storage): SSE-S3 (AES-256) / AWS-managed keys / ENFORCED (provider default)
   - Browser localStorage (JWT tokens): NOT ENCRYPTED / N/A / GAP -- see Section 5
   - Render application logs: Platform default / Render-managed / ENFORCED (provider default)
   - Render environment variables (secrets): Platform encryption / Render-managed / ENFORCED (provider default)

   Policy statement: ALL persistent data storage MUST use encryption at rest. Provider-managed encryption is the current standard.

5. **Key Management Principles (Section 4):**
   - Current approach: Infrastructure-provider-managed keys for all at-rest encryption (MongoDB Atlas, AWS S3, Render)
   - Application has NO application-level encryption key management
   - Application-level cryptographic operations:
     - Password hashing: bcrypt (adaptive, configurable cost factor) -- not encryption but relevant to data protection
     - Token generation: crypto.randomBytes for refresh tokens and invitation tokens
     - JWT signing: HMAC-SHA256 with server-held secret (JWT_SECRET, JWT_REFRESH_SECRET)
   - Key rotation:
     - Infrastructure keys: Managed by providers (Atlas, AWS, Render) -- transparent to application
     - JWT secrets: Manual rotation via Render environment variables. CRITICAL: rotation invalidates ALL active sessions (no dual-key validation window exists)
     - No automated key rotation mechanism exists (gap)
   - Secret storage: All secrets stored in Render environment variables (see BACK-01 Section 7 for full inventory)

6. **Gap Analysis (Section 5):**
   Follow the three-part pattern: Current State -> Gap -> Planned Remediation.

   Gap 1: No field-level encryption for RESTRICTED data
   - Current: Blob fields in import_log (previewData), deletion_snapshots (snapshotData, collectionSnapshots) contain RESTRICTED-equivalent data but use only Atlas-level encryption
   - Gap: Regulation 14 and data classification in DBDF-01 suggest RESTRICTED data should have additional protection beyond provider-level encryption
   - v1.6: Evaluate field-level encryption for blob collections or TTL-based auto-deletion as alternative mitigation

   Gap 2: No application-level encryption key rotation
   - Current: JWT secret rotation causes platform-wide session invalidation
   - Gap: No mechanism for graceful key rotation (dual-key validation window)
   - v1.6: Implement dual-key validation for JWT to allow rotation without mass session invalidation

   Gap 3: JWT stored in plaintext in browser localStorage
   - Current: JWT access and refresh tokens stored in localStorage, accessible via JavaScript (XSS vulnerability)
   - Gap: Regulation 12 and 14 imply data should be protected on user devices; localStorage provides no encryption
   - v1.6: Evaluate httpOnly cookie storage for tokens (reference R-04 in RISK-01)
   - Mitigation in place: 15-minute access token expiry limits exposure window

   Gap 4: Data classification handling rules NOT technically enforced by tier
   - Current: SECURITY-PROCEDURES.md Section 6 documents handling rules per classification tier, but no technical enforcement differentiates treatment of PUBLIC vs. RESTRICTED data
   - Gap: A sophisticated encryption policy would encrypt RESTRICTED fields with application-managed keys and PUBLIC fields with only transport encryption
   - v1.6: Evaluate field-level encryption for SENSITIVE/RESTRICTED classifications

   Gap 5: No certificate pinning or custom CA verification
   - Current: Relies on standard CA trust chain for all TLS connections
   - Gap: Provider certificate changes are transparent to the application -- no additional verification
   - Assessment: ACCEPTABLE risk for current security level (medium). Certificate pinning is a defense-in-depth measure appropriate for high-security databases.

7. **Encryption by Data Classification (Section 6):**
   Reference DATA-INVENTORY.md (DBDF-01) classification tiers and map to encryption standards:
   - PUBLIC: TLS in transit (enforced) / Atlas AES-256 at rest (enforced) / No additional encryption required
   - INTERNAL: TLS in transit (enforced) / Atlas AES-256 at rest (enforced) / No additional encryption required
   - SENSITIVE: TLS in transit (enforced) / Atlas AES-256 at rest (enforced) / Field-level encryption recommended for v1.6
   - RESTRICTED: TLS in transit (enforced) / Atlas AES-256 at rest (enforced) / Field-level encryption REQUIRED for v1.6

   Note: Currently ALL tiers receive identical encryption treatment. Differentiation is a v1.6 enhancement.

8. **Review Schedule (Section 7):** Annual review or upon changes to encryption infrastructure, provider changes, or security incidents involving data exposure. Must be reviewed before any new provider integration.

9. **Document Cross-References (Section 8):** Table of related documents.

**Anti-patterns to avoid:**
- Do NOT claim encryption capabilities beyond what ARCHITECTURE-DIAGRAM.md and DATA-FLOW-MAP.md document.
- Do NOT describe field-level encryption as if it exists today.
- Do NOT omit the localStorage JWT gap -- it is a known risk (R-04).
- Start from the existing encryption inventory in ARCHITECTURE-DIAGRAM.md Section 3 component table and Section 4 security boundary annotations, and DATA-FLOW-MAP.md Sections 4-5, then identify gaps. Do NOT write requirements first and then try to match them to reality.
