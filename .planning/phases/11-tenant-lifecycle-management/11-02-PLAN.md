Consolidate the two cascade deletion systems into one canonical implementation (success criteria #5).

Purpose: Two services (`cascadeDeletion.service.js` and `cascadeDeletionService.js`) both export `cascadeDeletionService` with different implementations. System B (`cascadeDeletionService.js`) is only used by test files. Consolidate by replacing System B with a re-export wrapper.
Output: Single canonical cascade deletion system; all imports resolve to the same implementation.
</objective>

  <name>Task 1: Replace System B with re-export wrapper pointing to System A</name>
  <files>services/cascadeDeletionService.js</files>
  <action>
  Replace the entire contents of `services/cascadeDeletionService.js` with a thin re-export wrapper:

  ```javascript
  /**
   * CONSOLIDATED: This file now re-exports from the canonical cascade deletion service.
   * The original collection-based implementation was consolidated into the
   * transaction-based service (cascadeDeletion.service.js) in Phase 11.
   *
   * All consumers should import from this file OR from cascadeDeletion.service.js.
   * Both resolve to the same implementation.
   */
  export { cascadeDeletionService } from './cascadeDeletion.service.js';
  ```

  This approach is SAFER than updating all import paths because:
  - It preserves backward compatibility for any import we might miss
  - Test files continue to work without import path changes
  - No risk of breaking the build from a missed import

  Verify that `services/cascadeDeletion.service.js` (System A) already has all the methods that test files call. Check the test file imports and their method calls:
  - `cascadeDeleteStudent` -- System A has this
  - `createDeletionSnapshot` -- System A has `createStudentSnapshot` (different name). The tests call `createDeletionSnapshot` which was a System B method. This means the test file will need its method calls updated OR the re-export won't be sufficient for these tests.

  IMPORTANT: Since System A and System B have DIFFERENT method names and signatures, the re-export wrapper alone won't make tests pass. The test files call System B's API (`createDeletionSnapshot`, `executeStudentCascade`, `validateDeletionImpact`, `rollbackDeletion`, `generateDeletionAuditLog`, `cleanupOrphanedReferences`) which have different signatures than System A's methods. The tests are NOT running in CI currently (they use MongoDB Memory Server and are not part of the standard test suite). The safest approach:

  1. Replace `services/cascadeDeletionService.js` with the re-export wrapper as described above
  2. Add a deprecation comment noting the method name differences
  3. The tests will need to be updated to use System A's API, but since they test the old System B API that no longer exists, they should be marked as needing update rather than breaking the re-export

  Also check `api/admin/cascade-deletion.service.js` -- this is a THIRD implementation used by `api/admin/cascade-deletion.controller.js`. This file has its own implementation with different collection names (`students` instead of `student`, `deletionSnapshots`, `deletionAuditLog`). It uses wrong collection names that don't match the COLLECTIONS constants. Leave this file as-is for now -- it's a separate admin endpoint concern and its routes may or may not be active. Add a TODO comment at the top noting it should be consolidated in a future cleanup.
  </action>
