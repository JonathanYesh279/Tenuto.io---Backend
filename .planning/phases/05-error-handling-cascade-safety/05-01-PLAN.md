<objective>
Sanitize all error responses to prevent cross-tenant information leakage.

Purpose: Error messages currently leak entity IDs and types to clients. The IDOR check in student.controller.js returns 403 (confirming a resource exists in another tenant). The global error handler passes raw err.message to clients, including internal details on 500 errors. All of these patterns must be standardized to return generic messages that never differ based on tenant boundaries.

Output: NotFoundError utility class, sanitized error handler, IDOR fix, controller-level error message cleanup across 6 controller files.
</objective>

  <name>Task 1: Add NotFoundError class and sanitize error handler</name>
  <files>utils/queryScoping.js, middleware/error.handler.js</files>
  <action>
  1. In utils/queryScoping.js, add a NotFoundError class AFTER the existing exports:

  ```javascript
  export class NotFoundError extends Error {
    constructor(resourceType = 'Resource') {
      super(`${resourceType} not found`);
      this.name = 'NotFoundError';
      this.statusCode = 404;
    }
  }
  ```

  Note: The constructor accepts resourceType for SERVER-SIDE logging only. The error handler must NOT pass this to the client. It should always return 'The requested resource was not found'.

  2. In middleware/error.handler.js, update the error handler:

  a) Update the 'NotFoundError' case (line 95-99) to return a generic message instead of err.message:
  ```javascript
  case 'NotFoundError':
    return res.status(404).json({
      error: 'Not Found',
      message: 'The requested resource was not found'
    })
  ```

  b) Update the statusCode === 404 handler (lines 103-107) to also use generic message:
  ```javascript
  if (err.statusCode === 404) {
    return res.status(404).json({
      error: 'Not Found',
      message: 'The requested resource was not found'
    })
  }
  ```

  c) Update the default 500 handler (lines 131-135) to NOT expose err.message:
  ```javascript
  return res.status(500).json({
    error: 'Internal Server Error',
    message: 'An unexpected error occurred'
  })
  ```

  Keep the console.error('Error:', err) at line 31 -- we still need server-side logging.

  d) For 403 responses (lines 110-115), keep err.message for legitimate role-based checks. These are NOT the same as IDOR leaks -- they indicate the user is authenticated but lacks permission for an operation they know about (e.g., admin-only action). The IDOR fix is handled separately in Task 2.

  e) For validation errors (400 status patterns), keep err.message since validation details are user-facing and expected.
