<task type="auto">
  <name>Task 2: Expand calculateStudentChanges to detect 7+ fields</name>
  <files>api/import/import.service.js</files>
  <action>
Replace the `calculateStudentChanges(student, mapped)` function (currently lines 1320-1338) with an expanded version that detects changes in all enriched fields. The new implementation must:

1. Keep the existing 3 flat field comparisons (studyYears, extraHour, class) -- these work correctly.

2. Add instrument name comparison:
   - Get the primary instrumentProgress entry from the existing student: find the entry with `isPrimary: true`, or fall back to index [0], or null if array is empty/missing.
   - Compare `mapped.instrument` against `primaryProgress?.instrumentName`. If both exist and differ, push a change with `field: 'academicInfo.instrumentProgress[0].instrumentName'`.

3. Add ministry stage level comparison:
   - Compare `mapped.ministryStageLevel` against `primaryProgress?.ministryStageLevel`. If mapped value exists and differs, push a change with `field: 'academicInfo.instrumentProgress[0].ministryStageLevel'`.

4. Add lesson duration comparison:
   - Compare `mapped.lessonDuration` (number, already converted by validateStudentRow) against `student.academicInfo?.lessonDuration`. If both exist and differ, push a change with `field: 'academicInfo.lessonDuration'`.

5. Add isBagrutCandidate comparison:
   - Compare `mapped.isBagrutCandidate` against `student.academicInfo?.isBagrutCandidate ?? null`. Only compare when `mapped.isBagrutCandidate` is not null and not undefined. Use strict equality (`!==`), NOT string coercion (both are booleans). Push change with `field: 'academicInfo.isBagrutCandidate'`.

6. Add teacher name comparison (display only for Phase 16):
   - If `mapped.teacherName` exists and is non-empty, push a change with `field: 'teacherName'`, `oldValue: null` (Phase 17 will resolve this), `newValue: mapped.teacherName`. This is always shown as a new value since we cannot match teachers yet. This gives the user visibility that teacher data exists in the import.

7. Add a special case for when the existing student has NO instrumentProgress array (empty or missing) but the import HAS instrument data. In this case, instead of comparing against a null primary, push a change with `field: 'academicInfo.instrumentProgress'`, `oldValue: null`, `newValue: mapped.instrument` to indicate "new instrument progress will be created".

IMPORTANT: The `field` values used in changes must match what Plan 02's execute logic will use to build MongoDB update operations. The `academicInfo.instrumentProgress[0].*` naming convention is critical -- Plan 02 will convert `[0]` to `.0.` for MongoDB dot notation.
