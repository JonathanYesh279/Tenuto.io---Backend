<task type="auto">
  <name>Task 1: Create roles service with permission customization and safeguards</name>
  <files>api/settings/roles.service.js</files>
  <action>
Create `api/settings/` directory and `roles.service.js` exporting `rolesService` with three functions:

1. `getRolePermissions(tenantId)` -- Returns the tenant's current rolePermissions (from DB) merged with DEFAULT_ROLE_PERMISSIONS as fallback. Also returns a list of all teachers in the tenant with their roles and coordinatorDepartments (projection: `_id, personalInfo.firstName, personalInfo.lastName, roles, coordinatorDepartments, isActive`).

2. `updateRolePermissions(tenantId, roleName, permissions)` -- Updates a single role's permissions in the tenant's rolePermissions document.
   - Validate `roleName` is in TEACHER_ROLES (import from constants.js). Return error if not.
   - **SAFE-02 enforcement**: If `roleName` is in ADMIN_TIER_ROLES, reject with error `"Cannot modify admin-tier role permissions"`. Admin permissions are always the frozen ADMIN_PERMISSIONS from defaults -- they cannot be customized.
   - For non-admin roles: validate `permissions` object using `validateRolePermissions({ [roleName]: permissions })`. If invalid, return the validation errors.
   - **LOCKED_DOMAINS enforcement**: The `validateRolePermissions` function already checks this, but double-check: if the role is not admin-tier and `permissions` contains `settings` or `roles` domains, reject.
   - On success: load current tenant rolePermissions (or deep-clone DEFAULT_ROLE_PERMISSIONS if null), set `rolePermissions[roleName] = permissions`, save via `updateOne({ _id }, { $set: { rolePermissions, updatedAt: new Date() } })`.
   - Return `{ success: true, roleName, permissions }`.

3. `resetRolePermissions(tenantId, roleName)` -- Resets a single role back to DEFAULT_ROLE_PERMISSIONS.
   - Validate `roleName` is in TEACHER_ROLES.
   - If admin-tier role: return success immediately (admin permissions are always defaults anyway).
   - Load current tenant rolePermissions. Set `rolePermissions[roleName]` to a deep clone of `DEFAULT_ROLE_PERMISSIONS[roleName]`. Save via updateOne.
   - Return `{ success: true, roleName, permissions: DEFAULT_ROLE_PERMISSIONS[roleName] }`.

Import from: `config/permissions.js` (DEFAULT_ROLE_PERMISSIONS, validateRolePermissions, LOCKED_DOMAINS), `config/constants.js` (TEACHER_ROLES, ADMIN_TIER_ROLES), `services/mongoDB.service.js` (getCollection), `mongodb` (ObjectId).

Use `createLogger('roles.service')` for logging.

<task type="auto">
  <name>Task 2: Create roles route and controller, mount in server.js</name>
  <files>api/settings/roles.controller.js, api/settings/roles.route.js, server.js</files>
  <action>
**roles.controller.js**: Export `rolesController` with three handlers:

1. `getRoles(req, res, next)` -- Calls `rolesService.getRolePermissions(req.context.tenantId)`. Returns `{ success: true, data: result }`.

2. `updateRole(req, res, next)` -- Reads `req.params.roleName` (URL-decoded automatically by Express). Calls `rolesService.updateRolePermissions(req.context.tenantId, roleName, req.body)`. On validation errors, return 400 with `{ success: false, error, errors }`. On success, return 200 with `{ success: true, data }`.

3. `resetRole(req, res, next)` -- Reads `req.params.roleName`. Calls `rolesService.resetRolePermissions(req.context.tenantId, roleName)`. On error return 400. On success return 200 with `{ success: true, data }`.

All handlers wrap in try/catch, pass errors to `next(err)`.

**roles.route.js**: Express router with three routes:
- `GET /` -- `requirePermission('roles', 'view')` -> `rolesController.getRoles`
- `PUT /:roleName` -- `requirePermission('roles', 'assign')` -> `rolesController.updateRole`
- `POST /:roleName/reset` -- `requirePermission('roles', 'assign')` -> `rolesController.resetRole`

Import `requirePermission` from `../../middleware/auth.middleware.js`.

**server.js**: Add import for `rolesRoutes` from `./api/settings/roles.route.js`. Mount it with the same middleware chain as other tenant-scoped routes:
```
app.use('/api/settings/roles', authenticateToken, enrichImpersonationContext, buildContext, enforceTenant, stripTenantId, rolesRoutes);
```
Place this BEFORE the catch-all/error handlers but alongside other `/api/` routes. No `addSchoolYearToRequest` needed (roles are not school-year scoped).
