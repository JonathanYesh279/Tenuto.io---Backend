**In rehearsal.service.js Рђћ rewrite `updateAttendance` function:**

Replace the existing non-transactional implementation at ~line 902. The current code does a `findOneAndUpdate` on rehearsal.attendance, then inside a try/catch does `activityCollection.deleteMany(...)` + individual `insertOne` calls Рђћ but the try/catch only logs errors without failing, so activity_attendance can silently fail while rehearsal.attendance succeeds. This entire block must be replaced with the new transactional flow below.

The new flow:
1. Validate input using updated `validateAttendance` (expects `{ records: [{ studentId, status, notes }] }`)
2. Load rehearsal (existing `getRehearsalById`)
3. Load orchestra by `rehearsal.groupId` to get `memberIds`
4. **Membership validation**: Check every `record.studentId` is in `orchestra.memberIds`. If ANY student is not a member, throw an error with code `MEMBERSHIP_VALIDATION` and include the list of invalid student IDs. Do NOT proceed with partial writes.
5. **Transactional write using `withTransaction`**:
   a. Delete existing activity_attendance records for this sessionId + activityType + tenantId (within session)
   b. Insert new activity_attendance records Рђћ one per student record:
      ```js
      {
        studentId: record.studentId,
        activityType: rehearsal.type, // 'ОфОќОъОЋОеОф' or 'ОћОеОЏОЉ'
        groupId: rehearsal.groupId,
        sessionId: rehearsalId,
        date: rehearsal.date,
        status: record.status, // 'ОћОњОЎОб/Оћ', 'ОюОљ ОћОњОЎОб/Оћ', or 'ОљОЎОЌОЋОе'
        notes: record.notes || '',
        teacherId: teacherId,
        tenantId,
        createdAt: toUTC(now()),
      }
      ```
   c. Update rehearsal.attendance cache (denormalized):
      ```js
      {
        present: records.filter(r => r.status === 'ОћОњОЎОб/Оћ').map(r => r.studentId),
        absent: records.filter(r => r.status === 'ОюОљ ОћОњОЎОб/Оћ').map(r => r.studentId),
        late: records.filter(r => r.status === 'ОљОЎОЌОЋОе').map(r => r.studentId),
      }
      ```
   d. All three operations (delete old, insert new, update cache) use the `{ session }` option from withTransaction.

6. Remove the old try/catch around activity_attendance that logged but didn't fail Рђћ the transaction now guarantees atomicity. If activity_attendance write fails, the whole transaction rolls back.

7. Keep the authorization check (admin or conductor) as-is.
