**Section 2: Access Management (SECPR-01a)**
Each subsection uses the Current State / Gap / Planned Remediation pattern:

2.1 User Account Lifecycle:
- Current State: Teacher accounts created by tenant admin via admin dashboard or bulk import. Account includes roles[] array, credentials.email, credentials.password (bcrypt). Account deactivation via `isActive: false`.
- Gap: No formal account provisioning approval workflow. No automated deprovisioning for inactive accounts. No periodic access review process.
- Planned Remediation (v1.6): Account review mechanism, inactive account deactivation policy.

2.2 Role Assignment:
- Current State: Roles assigned during account creation or updated by tenant admin. Stored in teacher.roles[] array. Role changes take effect on next authentication (new JWT).
- Gap: No role change audit trail. No approval workflow for elevated roles (admin). No periodic role review.
- Planned Remediation (v1.6): Role change logging, admin role approval workflow.

2.3 Tenant Isolation:
- Current State: 5-layer defense documented in research. enforceTenant middleware, buildContext adds tenantId, stripTenantId prevents client override, buildScopedFilter injects tenantId into queries, requireTenantId guard in service layer.
- Gap: No automated tenant isolation testing. Isolation correctness depends on developer discipline in every new query.
- Planned Remediation (v1.6): Automated tenant isolation integration tests.

2.4 Super Admin Access:
- Current State: Separate super_admin collection, separate middleware (super-admin.middleware.js), `type: 'super_admin'` JWT claim. Can impersonate tenant users. Mutating actions during impersonation logged in platform_audit_log.
- Gap: Super admin seed endpoint has no rate-limit or environment guard. No MFA for super admin. No super admin access review process.
- Planned Remediation (v1.6): Rate-limit seed endpoint, super admin MFA, access review process.

**Section 3: Authentication Procedures (SECPR-01b)**
Each subsection uses the Current State / Gap / Planned Remediation pattern:

3.1 Authentication Architecture:
- Current State: JWT-based authentication. Access tokens (1h expiry, HS256). Refresh tokens (30d expiry). Token revocation via tokenVersion counter on teacher credentials. Tenant deactivation check blocks requests from deactivated tenants.
- Reference: ARCHITECTURE-DIAGRAM.md (SMAP-01) for component diagram

3.2 Password Management:
- Current State: bcrypt with 10 salt rounds. `requiresPasswordChange` flag for forced password change. checkPasswordChangeRequired middleware enforces the flag.
- Gap: Default password "123456" auto-set for teachers without password (R-05 from RISK-01). No password complexity requirements. No password history tracking. No password expiry policy.
- Planned Remediation (v1.6): Remove default passwords -- implement invitation flow. Add minimum password length and complexity. Reference RISK-01 R-05 specifically.

3.3 Token Management:
- Current State: HS256 symmetric JWT signing. Tokens stored in localStorage on frontend (XSS vector). Token includes tenantId, userId, role claims.
- Gap: No key rotation mechanism. Symmetric keys mean any party with the key can forge tokens. localStorage is vulnerable to XSS. No token binding.
- Planned Remediation (v1.6): Evaluate RS256 asymmetric signing, key rotation schedule, httpOnly cookie storage.

3.4 Session Controls:
- Current State: Stateless JWT-based sessions. No server-side session store. Access token 1h expiry provides implicit session timeout.
- Gap: No explicit session timeout. No concurrent session limits. No "force logout all sessions" capability (beyond tokenVersion increment).
- Planned Remediation (v1.6): Evaluate session management improvements.

3.5 Account Lockout:
- Current State: No account lockout mechanism. Failed login attempts are not tracked or limited.
- Gap: Brute force attacks are not mitigated at the application level.
- Planned Remediation (v1.6): Implement lockout after N failed attempts with progressive delay.

**Section 4: Authorization Procedures (SECPR-01c)**
Each subsection uses the Current State / Gap / Planned Remediation pattern:

4.1 Role-Based Access Control (RBAC):
- Current State: permissionService.js defines ROLE_PERMISSIONS for 5 roles (Admin, Teacher, Conductor, Ensemble Instructor, Theory Teacher). requireAuth middleware enforces role checks at route level.
- Gap: 3 declared roles lack formal RBAC definitions: Deputy Admin (Sgan Menahel), Accompanist (Melaveh/Moreh-Melaveh), Guest (Oreah). These are enforced via route-level requireAuth checks but have no explicit permission map.
- Planned Remediation (v1.6): Define ROLE_PERMISSIONS for all 8 roles. Audit route-level auth for consistency.

4.2 IDOR Prevention:
- Current State: canAccessStudent() in queryScoping.js checks student access against _studentAccessIds. canAccessOwnResource() validates resource ownership. buildScopedFilter() injects tenant and teacher scope.
- Gap: Not all endpoints use the queryScoping utilities consistently. Some rely on manual checks.
- Planned Remediation (v1.6): Audit all endpoints for consistent IDOR protection.

4.3 Minors' Data Access:
- Current State: Student data (minors under 18) is classified RESTRICTED per DATA-INVENTORY.md. Access scoped by teacher assignment. No separate access tier for minors' data versus adult data.
- Gap: No additional access controls for minors' data beyond standard teacher scoping. No minors' data access logging. No parental consent mechanism. Reference MINORS-DATA.md (DBDF-03) for 5 identified gaps.
- Planned Remediation (v1.6): Minors' data access logging, consent mechanism evaluation.

4.4 Principle of Least Privilege:
- Current State: Teachers see only their assigned students. Admins see all within their tenant. Super admins have cross-tenant access via impersonation.
- Gap: No formal least-privilege review process. Permissions are not regularly audited.
- Planned Remediation (v1.6): Periodic access review process.

**Section 5: Backup and Recovery Procedures (SECPR-02a)**
Each subsection uses the Current State / Gap / Planned Remediation pattern:

5.1 Backup Strategy:
- Current State: MongoDB Atlas provides automated managed backups (continuous backup with point-in-time recovery). Application-level backups via snapshot collections: deletion_snapshots (entity-level recovery), tenant_deletion_snapshots (tenant-level recovery), migration_backups (pre-migration state).
- Gap: Atlas backup configuration not formally documented or verified. No backup restoration testing schedule. No documented verification that backups are actually occurring.
- Planned Remediation (v1.6): Document Atlas backup configuration, implement quarterly backup restoration tests, backup verification monitoring.

5.2 Recovery Objectives:
- Current State: No formally documented Recovery Point Objective (RPO) or Recovery Time Objective (RTO).
- Recommended Defaults (subject to Security Officer approval):
  - RPO: 24 hours (maximum acceptable data loss)
  - RTO: 4 hours (maximum acceptable downtime)
  - Rationale: Pre-launch SaaS with zero production tenants. These targets are reasonable for a conservatory management platform where data changes are primarily during business hours.
- Gap: RPO/RTO are recommendations, not formally approved targets.
- Planned Remediation (v1.6): Formal RPO/RTO approval by Security Officer, disaster recovery testing.

5.3 Disaster Recovery:
- Current State: No formal disaster recovery plan. Render provides infrastructure redundancy. Atlas provides database redundancy.
- Gap: No documented recovery procedure for: complete database corruption, hosting provider outage, credential compromise requiring full key rotation.
- Planned Remediation (v1.6): Create disaster recovery runbooks for each scenario.

5.4 Business Continuity:
- Current State: No formal business continuity plan. Platform is a web application -- if hosting is available, the application is available.
- Gap: No documented procedures for: prolonged outage communication to tenants, degraded-mode operation, data breach notification (see Phase 29 Incident Response).
- Planned Remediation: Phase 29 (Incident Response Plan) and v1.6 (technical controls).

**Section 6: Data Handling and Retention (SECPR-03a)**
Each subsection uses the Current State / Gap / Planned Remediation pattern:

6.1 Data Classification and Handling:
- Current State: Four-tier sensitivity classification defined in DATA-INVENTORY.md (DBDF-01): PUBLIC, INTERNAL, SENSITIVE, RESTRICTED. 22 collections classified.
- Handling rules per classification:

| Classification | Access | Storage | Transmission | Disposal |
|---------------|--------|---------|-------------|----------|
| PUBLIC | No restrictions | Standard | Standard | No special requirements |
| INTERNAL | Authenticated users only | Standard MongoDB | TLS in transit | Standard deletion |
| SENSITIVE | Role-based, tenant-scoped | Encrypted at rest (Atlas) | TLS in transit | Secure deletion with audit trail |
| RESTRICTED | Role-based, tenant-scoped, access-logged | Encrypted at rest (Atlas) | TLS in transit | Secure deletion with snapshot + audit trail |

- Gap: Handling rules are documented here but NOT technically enforced differently by classification tier. All data gets the same technical treatment regardless of classification.
- Planned Remediation (v1.6): Implement classification-aware access controls, particularly access logging for RESTRICTED data.

6.2 Retention Policies:
- Current State: DATA-PURPOSES.md (DBDF-02) recommends retention periods for 11 collections. No retention policies are currently enforced. No TTL indexes exist on any collection (R-11 from RISK-01).
- Recommended retention periods (from DATA-PURPOSES.md, pending Security Officer approval):

| Collection Category | Recommended Retention | Lawful Basis |
|--------------------|-----------------------|-------------|
| Active records (teacher, student, orchestra) | Duration of enrollment + 2 years | Contractual obligation |
| Audit logs (deletion_audit, platform_audit_log, security_log) | 7 years | Legal obligation |
| Operational data (school_year, hours_summary) | Duration of active use + 1 year | Contractual obligation |
| Snapshots (deletion_snapshots, tenant_deletion_snapshots) | 90 days from creation | Legitimate interest (recovery) |
| Import logs (import_log with previewData) | 90 days from creation | Legitimate interest (import recovery) |

- Gap: Retention periods are recommendations, not binding policy. No automated enforcement. Snapshot data accumulates indefinitely.
- Planned Remediation (v1.6): TTL indexes on snapshot and import_log collections. Formal retention policy approval by Security Officer. Automated purge jobs.

6.3 Data Deletion Procedures:
- Current State: Three deletion mechanisms exist:
  1. Soft delete: `isActive: false` on teacher, student, orchestra
  2. Cascade deletion: Full entity removal with document snapshots in deletion_snapshots, orchestrated by cascadeDeletion.service.js
  3. Tenant purge: Complete tenant data removal with full dump in tenant_deletion_snapshots, with 90-day grace period
- Gap: Soft-deleted records remain queryable. No right-to-erasure (data subject deletion request) procedure. previewData in import_log not purged after import execution (R-06 from RISK-01). Deletion snapshots retained indefinitely (R-07 from RISK-01).
- Planned Remediation (v1.6): Data subject deletion request procedure, previewData auto-purge, snapshot TTL enforcement.
