**AUTH-POLICY.md (ACPOL-02):**

Document header:
- Document ID: ACPOL-02
- Title: "Password and Authentication Policy"
- Version: 1.0
- Date: 2026-03-02
- Classification: INTERNAL -- COMPLIANCE DOCUMENT
- Document Owner: Security Officer (SECOFF-01/02)
- Review Cycle: Annual or upon authentication system changes
- Related Documents: SECOFF-01/02, SECPR-01/02/03, RISK-01 (specifically R-04, R-05), ACPOL-01, GLOSS-01

Section 1: Purpose
Defines the authentication controls for all platform users per Regulations 13-14. Covers password management, token-based authentication, and session controls.

Section 2: Password Policy
Each area uses the Current State / Gap / Planned Remediation pattern:

| Area | Current Control | Gap | Planned v1.6 Hardening |
|------|----------------|-----|----------------------|
| Hashing algorithm | bcrypt, 10 salt rounds | None -- adequate | No change needed |
| Default passwords | "123456" auto-set for teachers without password | CRITICAL -- R-05 from RISK-01 | Remove default passwords; implement invitation flow with one-time setup token |
| Minimum length | No enforcement | No minimum length | Enforce 8+ characters minimum |
| Complexity rules | No enforcement | No complexity requirements | Require mixed case + number minimum |
| Password history | Not tracked | Users can reuse previous passwords | Track last 5 passwords, prevent reuse |
| Password expiry | No expiry policy | Passwords never expire | Evaluate 12-month expiry for admin roles |
| Forced change | requiresPasswordChange flag exists | Only used for initial setup | Extend to post-breach forced change |

Section 3: Token Management

3.1 JWT Architecture:
- Signing: HS256 (symmetric HMAC-SHA256)
- Access token: 1-hour expiry, contains userId, tenantId, role claims
- Refresh token: 30-day expiry, used to obtain new access tokens
- Revocation: tokenVersion counter on teacher credentials -- incrementing invalidates all existing tokens
- Super admin tokens: Contain `type: 'super_admin'` claim, validated by separate middleware
- Impersonation tokens: Contain `isImpersonation: true` claim, scoped to target tenant

3.2 Token Security Assessment:

| Aspect | Current State | Risk Level | v1.6 Plan |
|--------|--------------|-----------|-----------|
| Signing algorithm | HS256 symmetric | MEDIUM (R-04) | Evaluate RS256 asymmetric |
| Key rotation | No rotation mechanism | HIGH | Implement scheduled rotation |
| Token storage | localStorage (frontend) | MEDIUM (XSS vector) | Evaluate httpOnly cookies |
| Token binding | None | LOW | Evaluate device/IP binding |
| Refresh token scope | 30-day validity | MEDIUM | Evaluate shorter validity + rotation |

Section 4: Account Lockout Policy
- Current State: NOT IMPLEMENTED
- Recommended policy (v1.6):
  - Lock account after 5 consecutive failed login attempts
  - Progressive delay: 1min after 5, 5min after 8, 30min after 10
  - Admin unlock capability
  - Lockout events logged to security_log

Section 5: Multi-Factor Authentication
- Current State: NOT IMPLEMENTED for any role
- Recommended policy (v1.6):
  - Mandatory for super admin accounts
  - Recommended for tenant admin accounts
  - Optional for teacher accounts
  - Implementation: TOTP-based (app authenticator)

Section 6: Session Management
- Current State: Stateless JWT-based. No server-side session store. Implicit timeout via 1h access token expiry.
- Gap: No concurrent session limits, no "logout all devices" beyond tokenVersion bump, no session activity tracking
- v1.6 Plan: Evaluate session management improvements

Section 7: Known Authentication Risks
Reference RISK-ASSESSMENT.md (RISK-01) risks:
- R-04: JWT Secret Compromise -- mitigated by environment variable isolation, residual risk: key rotation needed
- R-05: Default Password Exploitation -- CRITICAL gap, remediation: invitation flow in v1.6

**ACCESS-LOGGING-POLICY.md (ACPOL-03):**

Document header:
- Document ID: ACPOL-03
- Title: "Access Logging Policy"
- Version: 1.0
- Date: 2026-03-02
- Classification: INTERNAL -- COMPLIANCE DOCUMENT
- Document Owner: Security Officer (SECOFF-01/02)
- Review Cycle: Annual or upon logging infrastructure changes
- Related Documents: SECOFF-01/02, SECPR-01/02/03, ACPOL-01, DBDF-01, DBDF-03, RISK-01 (R-08), GLOSS-01

Section 1: Purpose
Defines the access logging requirements per Regulation 12. Specifies what events must be logged, where logs are stored, retention periods, and the review process.

Section 2: Current Logging Infrastructure
Document what exists today:

| Log Target | Storage | What Is Logged | Scope |
|-----------|---------|---------------|-------|
| platform_audit_log | MongoDB collection | Super admin actions: impersonation start/end, tenant management, user management, mutating actions during impersonation | Platform-wide |
| deletion_audit | MongoDB collection | Cascade deletion events: entity type, entity IDs, operator, timestamp | Per-tenant |
| security_log | MongoDB collection | Limited security events (used sparingly) | Per-tenant |
| import_log | MongoDB collection | Import operations: file metadata, preview data, execution results | Per-tenant |
| Pino logger | stdout (Render logs) | Application-level operational logs: request/response, errors | Unstructured, not compliance-grade |

Section 3: Required Logging Events (Current State / Gap / v1.6 Plan)
Define event categories with logging requirements:

| # | Event Category | Specific Events | Current State | Priority | v1.6 Plan |
|---|---------------|----------------|---------------|----------|-----------|
| 1 | Authentication | Login success, login failure (with IP, user agent), logout, token refresh, password change | NOT LOGGED | CRITICAL | Structured auth event logging to security_log |
| 2 | Authorization failures | Permission denied events, role check failures, IDOR attempts | NOT LOGGED (silently rejected) | HIGH | Log to security_log with request context |
| 3 | Minors' data access | Read/write operations on student and bagrut collections | NOT LOGGED | CRITICAL (per DBDF-03) | Dedicated minors' data access log |
| 4 | Data modification | Create/update/delete on PII-containing collections (11 collections from DBDF-02) | PARTIAL -- deletion_audit covers deletes only | HIGH | Comprehensive change audit log |
| 5 | Admin operations | Tenant admin role changes, user creation, bulk operations | NOT LOGGED at tenant level | MEDIUM | Tenant admin audit log |
| 6 | Super admin operations | All platform management actions | LOGGED in platform_audit_log | ADEQUATE | Maintain current logging |
| 7 | Impersonation actions | All mutating actions during impersonation sessions | LOGGED in platform_audit_log | ADEQUATE | Maintain current logging |
| 8 | Data export events | Ministry report generation, Excel exports | PARTIAL -- snapshots saved but no event log | MEDIUM | Export event logging |
| 9 | Data import events | Import operations, preview generation | PARTIAL -- import_log metadata exists | LOW | Enhance with security event context |
| 10 | Configuration changes | School year changes, tenant settings modifications | NOT LOGGED | MEDIUM | Configuration change audit log |

Section 4: Log Entry Format
Define the required fields for each log entry:

| Field | Description | Example |
|-------|------------|---------|
| timestamp | ISO 8601 UTC | 2026-03-02T14:30:00.000Z |
| eventType | Category from Section 3 | "authentication.login_success" |
| actorId | User performing the action | ObjectId (teacher or super_admin) |
| actorRole | Role at time of action | "מנהל" |
| tenantId | Tenant context | ObjectId |
| resourceType | Target collection/entity | "student" |
| resourceId | Target entity ID | ObjectId |
| action | Specific operation | "read", "update", "delete" |
| details | Additional context | { fields_changed: [...] } |
| ipAddress | Client IP | "192.168.1.1" |
| userAgent | Client browser/app | "Mozilla/5.0..." |
| result | Outcome | "success" or "denied" |

Section 5: Log Retention Policy

| Log Category | Retention Period | Rationale | Storage |
|-------------|-----------------|-----------|---------|
| Authentication events | 2 years | Security investigation window | security_log collection |
| Authorization failures | 2 years | Security investigation window | security_log collection |
| Minors' data access | 7 years | Regulatory requirement for minors' data | Dedicated collection (v1.6) |
| Data modification audit | 7 years | Legal obligation | Dedicated collection (v1.6) |
| Admin operations | 7 years | Legal obligation | platform_audit_log / tenant audit log |
| Super admin operations | 7 years | Legal obligation | platform_audit_log |
| Impersonation actions | 7 years | Legal obligation | platform_audit_log |
| Export events | 7 years | Ministry reporting traceability | Dedicated collection (v1.6) |
| Import events | 2 years | Operational | import_log (existing) |
| Application logs (Pino) | 30 days | Operational debugging | Render log retention |

Section 6: Log Review Process

6.1 Scheduled Reviews:

| Review Type | Frequency | Reviewer | Focus |
|------------|-----------|---------|-------|
| Authentication anomalies | Weekly (when implemented) | Security Officer or delegate | Failed logins, unusual patterns, account lockouts |
| Minors' data access | Monthly (when implemented) | Security Officer | Verify access is appropriate and scoped |
| Admin action review | Monthly | Security Officer | Role changes, user management actions |
| Super admin activity | Monthly | Security Officer | Impersonation sessions, platform changes |
| Full audit log review | Quarterly | Security Officer | Comprehensive review across all categories |

6.2 Ad-hoc Review Triggers:
- Suspected security incident
- User complaint about unauthorized access
- Anomaly detected in scheduled review
- Regulatory audit request
- Data subject access request

6.3 Escalation Procedure:
- Anomaly detected -> Security Officer notified within 24 hours
- Confirmed security event -> Incident Response Plan activated (Phase 29)
- Regulatory breach suspected -> Senior management notified immediately

Section 7: Current Gaps and Remediation (R-08 from RISK-01)
Reference R-08 (Insufficient Logging for Compliance) from RISK-ASSESSMENT.md:
- CRITICAL gap: Categories 1-3 (authentication, authorization, minors' data access) are NOT logged at all
- HIGH gap: Category 4 (data modification) is only partially logged
- MEDIUM gap: Categories 5, 8, 10 have no logging
- ADEQUATE: Categories 6, 7 (super admin and impersonation) are already logged
- v1.6 roadmap: Implement structured logging for categories 1-5 minimum. Categories 8-10 as stretch goals.

Section 8: User Notification
- Note: Users should be informed that their access and activity are logged (full user notification policy is Phase 30 deliverable)
- Recommended: Add login banner or terms-of-service clause informing users of activity monitoring
