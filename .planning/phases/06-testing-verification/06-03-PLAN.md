**test/tenant-isolation/write-isolation.test.js** — Cross-tenant write rejection tests:

Use the same ESM-safe setup pattern from read-isolation.test.js (setupTenantIsolationDB -> patchMongoDBService -> dynamic imports in beforeAll).

**`describe('Cross-Tenant Write Isolation — Student')`:**
- `it('PUT /api/student/:id — Tenant A cannot update Tenant B student')`: Send PUT with tokenA to `/api/student/${studentB._id}` with body `{ personalInfo: { firstName: 'Hacked' } }`. Expect 404. Then query DB directly to confirm studentB.personalInfo.firstName is unchanged.
- `it('DELETE /api/student/:id — Tenant A cannot delete Tenant B student')`: Send DELETE with tokenA to `/api/student/${studentB._id}`. Expect 404. Query DB to confirm studentB still exists.

**`describe('Cross-Tenant Write Isolation — Teacher')`:**
- `it('PUT /api/teacher/:id — Tenant A cannot update Tenant B teacher')`: PUT with tokenA to `/api/teacher/${teacherB._id}`, expect 404. Verify DB unchanged.

**`describe('Cross-Tenant Write Isolation — Orchestra')`:**
- `it('PUT /api/orchestra/:id — Tenant A cannot modify Tenant B orchestra')`: PUT with tokenA, expect 404.

**`describe('Cross-Tenant Write Isolation — School Year')`:**
- `it('PUT /api/school-year/:id/set-current — Tenant A cannot set Tenant B school year as current')`: PUT with tokenA, expect 404.

**`describe('Cross-Tenant Write Isolation — Theory')`:**
- `it('PUT /api/theory/:id — Tenant A cannot update Tenant B theory lesson')`: PUT with tokenA, expect 404.

**`describe('Cross-Tenant Write Isolation — stripTenantId Enforcement')`:**
- `it('POST with mismatched tenantId in body returns 400 TENANT_MISMATCH')`: POST to `/api/student` with tokenA but body includes `tenantId: TENANT_B_ID`. Expect 400 with `error: 'TENANT_MISMATCH'`.

For each write test:
- Make the HTTP request via supertest
- Assert the response status (404 for cross-tenant resource, 400 for mismatch)
- IMPORTANT: After the request, query the DB directly to verify the target document was NOT modified/deleted. This is the belt-and-suspenders check.

**test/tenant-isolation/concurrent/async-context.test.js** — Concurrent request contamination test:

Use the same ESM-safe setup pattern (setupTenantIsolationDB -> patchMongoDBService -> dynamic imports).

**`describe('Async Context Contamination')`:**
- `it('120 concurrent requests from alternating tenants never leak data', async () => {...})`

Implementation:
1. Create tokenA and tokenB using makeToken.
2. Build request array of 120 entries, alternating: even indices use tokenA, odd use tokenB.
3. Fire ALL requests simultaneously using `Promise.all`:
   ```javascript
   const promises = requests.map(({ token, expectedTenantId }, i) =>
     request(app)
       .get('/api/student')
       .set('Authorization', `Bearer ${token}`)
       .then(res => ({ index: i, expectedTenantId, status: res.status, body: res.body }))
   );
   const results = await Promise.all(promises);
   ```
4. For each result:
   - Assert status is 200 (all requests should succeed).
   - Extract student array from response (handle both `res.body.data` and `res.body.students` and `Array.isArray(res.body)` response shapes).
   - For every student in the response, assert `student.tenantId === expectedTenantId`.
   - Count contaminations (wrong tenantId) separately so the error message is clear: `"Request ${i}: expected ${expectedTenantId} but got ${student.tenantId}"`.
5. Final assertion: `expect(contaminations).toBe(0)`.

**Additional test:**
- `it('60 concurrent writes to different tenants do not cross-contaminate', async () => {...})`
  - Fire 60 POST/PUT requests alternating between tenants (e.g., updating student names).
  - After all complete, query DB directly for each tenant's students.
  - Verify no student has been modified by the other tenant's request.
  - This is important because write paths may share module-level state (unlike reads which are query-only).

**Timeout:** Set `{ timeout: 60000 }` on the describe block since 120 concurrent requests through real MongoDB may be slow.

**Note:** The /api/student endpoint was chosen because it is the most commonly used and exercises the full middleware chain + buildScopedFilter + MongoDB query. If student listing requires schoolYearId, pass schoolYearA._id or schoolYearB._id as appropriate in the query string.
